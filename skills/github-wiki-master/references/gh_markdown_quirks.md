# GitHub Flavored Markdown (GFM) & Wiki Quirks

**TARGET AUDIENCE:** Automated Agents & AI Code Generators
**PURPOSE:** Strict structural constraints for generating GitHub Wiki compatible markdown. Violating these rules will result in broken rendering, dead links, or stripped content.

## 1. Link & Routing idiosyncrasies

* **Space-to-Hyphen Mapping:** GitHub Wikis internally map spaces in page titles to hyphens in URLs. `[[My Page]]` routes to `My-Page.md`. ALWAYS prefer hyphens in actual filenames to ensure deterministic resolution.
* **Case Sensitivity:** GitHub's backend rendering runs on Linux, making file resolution strictly case-sensitive. `[[api-reference]]` will NOT resolve to `API-Reference.md`. Match casing perfectly.
* **The Pipe Syntax:** When using Wiki links `[[ ]]`, the display text and target are separated by a pipe. **CRITICAL:** The display text is on the LEFT, target on the RIGHT `[[Display Text|Target-Page]]`. This is the exact opposite of standard markdown `[Display Text](Target-Page)`.
* **Anchor Tags:** Internal page anchors are generated by down-casing headers and removing punctuation. `## API Endpoints!` becomes `#api-endpoints`. Link to it via `[[Page-Name#api-endpoints]]`.

## 2. Asset & Image Management

* **No Direct Uploads via Git:** You cannot "upload" an image to a wiki page via standard markdown like you can via the GitHub Web UI.
* **Relative Pathing:** Images must be committed to the `.wiki.git` repository. Store them in an `images/` directory.
* **Syntax:** Reference them strictly via relative pathing: `[[/images/architecture.png]]` or `<img src="/images/architecture.png" width="500">`. Do not use absolute `https://raw.githubusercontent...` paths, as they will break if the repository becomes private.

## 3. HTML Sanitization (The Scrubbing Engine)

GitHub employs an aggressive HTML pipeline that sanitizes user input.

* **STRIPPED ENTIRELY:** `<script>`, `<iframe>`, `<link>`, `<style>`, `<applet>`, `<object>`.
* **CSS:** Inline styles (`style="..."`) are mostly stripped, except for safe properties like `color`, `text-align`, and `width`/`height`. NEVER rely on inline CSS for critical layout. Use standard markdown tables.
* **Line Breaks:** Standard Markdown requires two spaces at the end of a line for a break. GFM supports hard line breaks natively, but to ensure cross-parser safety, explicitly use `<br>` for forced vertical spacing inside tables or complex lists.

## 4. Advanced GFM Features

* **Math Formulas:** Use `$` for inline math and `$$` for block math (rendered via MathJax/KaTeX). Avoid using raw `<img>` tags for equations.
* **Task Lists:** Supported natively via `- [ ]` and `- [x]`. Always include a space after the bracket.
* **Alerts/Callouts:** GitHub supports blockquote-based alerts. Use them to highlight critical warnings to developers:

  ```markdown
  > [!NOTE]
  > Useful information.
  
  > [!WARNING]
  > Urgent developer warnings.
  ```

  *Supported tags: `[!NOTE]`, `[!TIP]`, `[!IMPORTANT]`, `[!WARNING]`, `[!CAUTION]`.*

## 5. Reserved Namespaces

* `_Sidebar.md`: Controls global right-hand navigation.
* `_Footer.md`: Injected at the bottom of all rendered pages.
* `Home.md`: The immutable landing page (`/wiki` root).
* **RULE:** Never use the `_` prefix for standard documentation pages.
